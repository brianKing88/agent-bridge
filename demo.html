Before vs After

现在：人肉中继

开发者打开两个终端
→ 在终端 A 看到后端 agent 的输出
→ 手动复制粘贴到终端 B
→ 前端 agent 看到后回复
→ 开发者再复制回终端 A

每一轮对话都需要人工介入。
开发者成了最慢的环节。

Agent Bridge：自动协作

一键启动 Hub + 两端 Plugin
→ 后端 agent 自动发消息给前端
→ 前端 agent 自动回复
→ 来回 N 轮，人类只需旁观

开发者可以随时插话干预。
但大多数时候，喝杯咖啡就好。

JSON-RPC 方法（4 个）

Plugin → Hub，HTTP POST /rpc

方法参数返回说明
registeragent_id, role, description{ ok: true }Agent 注册身份
list_agents（无）[{ agent_id, role, status }]查看谁在线
send_messageto, content{ id, timestamp }发消息（to="*" 广播）
read_messagessince_id?, from?, limit?[{ id, from, content, ts }]拉取消息（补漏）

SSE 事件（3 个）

Hub → Plugin，GET /events（实时推送）

事件数据触发时机
message{ id, from, to, content, timestamp }收到发给该 agent 的消息
agent_online{ agent_id, role, description }新 agent 上线
agent_offline{ agent_id }Agent 断线

+ 每 15 秒 SSE comment 心跳（: heartbeat）

Tool参数Agent 调用示例

认证

三步启动，零配置

1

任意一台机器上启动 Hub

npx agent-bridge hub

Hub 启动后打印 Token 和地址。所有 agent 连这个地址。

2

机器 A：后端 agent 加入

export AGENT_BRIDGE_TOKEN=xxx && export AGENT_BRIDGE_HUB=http://hub-ip:9900

在 Claude Code 的 MCP 配置中添加 agent-bridge plugin，重启 Claude Code。

3

机器 B：前端 agent 加入

export AGENT_BRIDGE_TOKEN=xxx && export AGENT_BRIDGE_HUB=http://hub-ip:9900

同样配置 MCP plugin。两个 agent 自动发现对方，开始协作。

背后发生了什么

每条消息的技术流程

1. 后端 Agent 调用 MCP Tool: send_message(to="frontend-web-01", content="...")
↓ Plugin 将 MCP tool call 转为 JSON-RPC
2. Plugin → Hub: POST /rpc {"method":"send_message","params":{"to":"frontend-web-01","content":"..."}}
↓ Hub 路由消息，通过 SSE 推送给目标 agent
3. Hub → 前端 Plugin: SSE event: message {"from":"backend-drone-01","content":"..."}
↓ Plugin 存入本地队列
4. 前端 Agent 调用 MCP Tool: read_messages() → 从本地队列拉取消息
↓ Agent 理解消息内容，决定下一步行动
5. 前端 Agent 修改代码，然后回复：send_message(to="backend-drone-01", content="已适配")
延迟：同局域网 < 10ms，跨公网 < 200ms
